# 5.1 System Path Mappings & Architecture Standards

Unlike Nginx, which combines public keys into a single bundle, Apache HTTPD traditionally processes the end-entity certificate and the intermediate authority bundle via separate operational directives.

### File Distribution Framework

On **RHEL, Rocky Linux, and Fedora** topologies, cryptographic assets must follow the standard Red Hat security pathing standards. Copy the assets generated by your automated Bash toolset into the corresponding authoritative system directory targets:

```bash
sudo cp srv-web01.local.key.pem  /etc/pki/tls/private/srv-web01.local.key
sudo cp srv-web01.cert.pem       /etc/pki/tls/certs/srv-web01.local.crt
sudo cp intermediate.cert.pem    /etc/pki/tls/certs/intermediate.crt
sudo cp ca.cert.pem              /etc/pki/tls/certs/ca.crt

```

### Applying Strict Access Control Lists (ACLs)

To pass SELinux and standard UNIX discretionary access controls, restrict the Private Key file permissions immediately to prevent low-privileged service hooks or unauthorized local users from extracting your operational cryptographic primitives:

```bash
sudo chmod 600 /etc/pki/tls/private/srv-web01.local.key
sudo chmod 644 /etc/pki/tls/certs/srv-web01.local.crt /etc/pki/tls/certs/intermediate.crt /etc/pki/tls/certs/ca.crt

# Ensure standard root ownership across the security boundaries
sudo chown -R root:root /etc/pki/tls/

```
